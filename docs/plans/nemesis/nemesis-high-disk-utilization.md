# Nemesis High Disk Utilization Support

## 1. Problem Statement

The 90%-disk elasticity test (`test-cases/features/elasticity/elasticity-90-percent-with-nemesis.yaml`)
selects nemesis with `nemesis_selector: 'supports_high_disk_utilization'`. Today **15 nemesis are
excluded** because they carry `supports_high_disk_utilization = False`. This shrinks chaos coverage
under the high-disk condition â€” the exact regime where ScyllaDB is most fragile (compaction, streaming,
and ENOSPC handling). Several exclusions are conservative or implementation-driven rather than
fundamental, so real bugs in those operations at 90% disk go untested.

Measurable pain points:
- 15 of the disruptions are unavailable at 90% disk (â‰ˆ the entire topology-shrink, MV/SI, backup, and
  scrub families).
- The flag was applied largely as a blanket "consumes disk â†’ off" with terse one-line comments; no
  systematic analysis distinguishes *inherent* limits from *fixable* ones.
- At least one exclusion (`#22088`) masks an open **ScyllaDB bug**, but the masking is permanent (no
  auto re-enable when the bug is fixed).

**Capacity reasoning (why grow-first works for the topology-shrink family):**
Cluster of `n` nodes at ~90%, dataset `D`, RF `r` â†’ per-node `â‰ˆ DÂ·r/n`. **Shrink-first** (current):
node dies â†’ `removenode` rebuilds its replicas onto `n-1` survivors â†’ `DÂ·r/(n-1) > 90%` â†’ ENOSPC.
**Grow-first** (proposed): add a node â†’ existing nodes stream out to it (usage drops) â†’ then
terminate + `removenode` â†’ back to `n` live nodes at `DÂ·r/n` â‰ˆ the steady 90% the cluster already
sustains. Safe.

## 3. Goals

1. Reduce the high-disk exclusion set from **15 â†’ â‰¤2** nemesis (only the `#22088`-blocked ones remain
   excluded, and only until the bug is fixed).
2. Enable the topology-shrink family (`TerminateAndRemoveNodeMonkey`, both `IsolateNode*`, optionally
   both `Decommission*`) to run at 90% via grow-before-shrink, with no node exceeding ~95% disk on the
   elasticity-90 run.
3. Enable MV/SI nemesis (`CreateIndexNemesis`, `AddRemoveMvNemesis`, `KillMVBuildingCoordinator`) by
   operating on a dedicated, row-bounded table whose index/MV fits within free headroom.
4. Enable Manager backup nemesis (`MgmtBackupSpecificKeyspaces`, `MgmtBackup`, `ManagerRcloneBackup`,
   `ManagerNativeBackup`) with a pre-flight free-space guard and prompt snapshot deletion.
5. Make `MgmtRestore` size the restored snapshot to *actual current free space*, or document it as
   inherent and keep it off.
6. Replace the permanent `#22088` exclusions with a `SkipPerIssues("scylladb/scylladb#22088")` gate so
   they auto-enable when the bug closes.
7. Every flag change is accompanied by regenerated `data_dir/nemesis.yml` + pipelines and a green
   elasticity-90 validation run.

## 4. Implementation Phases

> Default behavior must be preserved for normal longevity. Where a behavior change (e.g. grow-first)
> alters coverage, prefer a **dedicated high-disk variant class** or a **runtime disk-usage check**
> over unconditionally changing the existing nemesis. See Needs-Investigation notes.

## 7. Risk Mitigation
